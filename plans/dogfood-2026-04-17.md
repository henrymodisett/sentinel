## Open findings

### [8] Coder↔Reviewer iteration loop — FIXED (this branch)

**Rationale:** the whole point of Sentinel is to discover impactful work **and ship it**. Without an iteration loop, the first reviewer verdict is final — a coder producing partial/buggy work gets rejected once and the whole effort rots on an unmerged branch. Cycle 3 demonstrated this with 2/2 items rejected on fixable findings. The pipeline was "find good work, then fail to ship it."

**Implementation:** `_iterate_coder_reviewer` in `work_cmd.py` wraps the initial review. On non-approved verdict, re-invokes `Coder.execute` with `review_feedback=ReviewResult` so the coder gets the REVISE_PROMPT (explicit blocking issues, "refine don't rewrite" framing, preserved scope rules). Re-runs reviewer. Caps at `MAX_CODER_ITERATIONS = 3`.

**Termination conditions** (priority order):
1. `review.verdict == 'approved'` → proceed to verifier + ship
2. `iterations >= MAX` → stop, final verdict reported to cycle summary
3. **No-progress guard:** if two consecutive reviews produce the same (order-insensitive, whitespace-normalized) `blocking_issues` set, the coder isn't responding to feedback — stop burning budget
4. `exec_result.status == 'failed'` mid-loop → bail, surface the error

**Integration with Touchstone:** clean. The iteration loop is internal to Sentinel (coder ↔ sentinel's LLM reviewer, cheap + fast). Touchstone's Codex pre-push review still runs at push time on the final cleaned-up diff. Boundary from CLAUDE.md holds: no imports, no subprocess calls. Future: when Codex blocks a push, its findings could feed into another iteration pass — the `review_feedback` seam is abstract enough.

**Tests:** `tests/test_coder_iteration.py` — 10 cases covering approve-on-2nd-pass, approve-on-3rd, max-iterations-cap, no-progress-early-stop, order/whitespace-insensitive comparison, coder-failure-breaks-loop, REVISE_PROMPT rendering.

**Per-iteration cost:** each revise pass costs ~1 coder call + 1 reviewer call (~$0.50-$0.60 combined with current routing). Worst case: 3 iterations burns ~$1.80 per item. The no-progress guard cuts this by ~33% when the coder is stuck. This is still much cheaper than a merged PR + later revert, which is the alternative.

### [7] End-to-end pipeline works; coder quality is the gate — T1 IN PROGRESS

**Observation:** cycle 3 completed cleanly — 2 work items processed, each through coder → reviewer. Reviewer correctly rejected both (item 1: coder only addressed 2 of 10 target files and hardcoded image dimensions; item 2: coder emitted invalid CSS `@src/app/components/MediaItem.tsx (prefers-reduced-motion: reduce)` instead of `@media (...)`).

**What this proves:** harness and safety rail are sound. Reviewer catches real bugs; no bad PRs shipped. Scan + plan + execute + review + gate all compose.

**What it exposes:** the bottleneck for the "autonomous PR factory" vision is **coder output quality**, not the harness. Two independent failure modes:
- **Scope discipline** — coder declared done after touching 2/10 files named in the work item's "files likely to be touched"
- **Basic syntax correctness** — coder emitted invalid CSS that the claude-sonnet-4-6 model should have caught in its own test pass

**Proposed directions (not blocking for ship, but shape v0.2):**
- Have the coder prompt mandate a "acceptance checklist" that enumerates each file in "files likely to be touched" and verifies each was modified (or explicitly justified as not needing change).
- Run the target project's lint/format hook inside the coder's verification step, before the reviewer sees the diff. A CSS syntax error should fail `next lint` / `prettier` locally and force the coder to retry.
- Consider a "syntax sanity" reviewer lens that runs cheap static checks (ESLint, CSS parser, etc.) before invoking the LLM reviewer — would have caught the `@src/app/...` bug for ~free.

**Has NOT been proven end-to-end yet:** push-recovery path in a real cycle. Neither rejected item tried to push. Unit tests cover it; need a cycle where an item passes review to prove the full happy path.

**Tier breakdown (this PR addresses T1):**
- **T1 — Planner emits sharp work items (this branch).** `SYNTHESIZE_SCHEMA` now requires `acceptance_criteria`, `verification`, `out_of_scope` per top_action; `files` is now `[{path, rationale}]`. `SYNTHESIZE_PROMPT` instructs the LLM to populate them with high-signal content. Scan markdown writer + `_parse_actions_from_scan` + `_write_backlog` all carry the new fields end-to-end. Backwards-compat: legacy flat `**Files:** a, b, c` still parses. Tests in `tests/test_plan_cmd.py` (16 cases) cover parser, renderer, and full round-trip. **Hypothesis test #2 of "sharp work items → mergeable code":** this PR was itself dispatched as a sharp hand-written work item to a sentinel-coder agent — produced clean code on first pass, 396/396 tests pass, ruff clean. Two for two on the hypothesis.
- **T2 — Coder consumes the new fields explicitly (next PR).** `BUILD_PROMPT` references `acceptance_criteria` / `verification` / `out_of_scope` directly; coder must run `verification` commands and report green before declaring done.
- **T3 — Reviewer checks against acceptance criteria (follow-up).** Reviewer takes the work item's `acceptance_criteria` and explicitly checks each one against the diff before issuing a verdict.

### [6] Cycle summary miscounts review-rejected items as neither approved nor failed — FIXED (this branch)

**Symptom:** after cycle 3, the final banner showed `Items executed: 2 • Approved: 0 • Failed: 0`. Both items were coded, reviewed, and rejected — they vanished from the totals because the counter only incremented on `success == "approved"` or `success == "failed"`. `rejected` and `changes` (changes-requested) slipped through both branches.

**Fix:** introduced a third bucket. `_bucket_outcome(success)` maps `_execute_and_review`'s success token to one of approved/rejected/failed and is exhaustive (any unexpected token buckets to failed so it stays visible). Summary panel now shows `Approved: N / Rejected: N / Failed: N` and the three sum to executed.

**Why a separate bucket (not lumped with failed):** rejection means coder quality / scope was the gate (try iterating, tighten the coder prompt, switch model). Failure means the harness itself broke (look at the journal). Operators need to distinguish those — especially now that the iteration loop (PR #63) tries multiple times before reporting `rejected`. A cycle reporting "1 rejected" is a different intervention from "1 failed."

**Regression tests:** `tests/test_cycle_summary.py` — 9 cases covering each known token, defensive bucketing of unexpected/empty/None tokens, the partition invariant (sum of buckets == executed), and the rejected-vs-failed distinction.

### [5] `--budget` confusion compounds the user's trust in the cost gate — FIXED (this branch)

**Related to [4].** After I hit #4, the instinct was "just bump the budget" — which works but defeats the purpose. A new user would probably do the same, then burn more than intended on the next run before figuring out the semantics.

**Root cause beyond [4]:** the gate was opaque in-flight. Users could see `Spend today` in the final Panel but couldn't see their *current cycle's* spend accumulating in real time. So `--budget $5` was a black box that either let you finish or blocked with a terse message — no way to feel when you were 80% of the way there.

**Fix:** live cycle-spend visibility, shown only when `--budget $X` includes a money cap. `_format_cycle_spend_line(today_spent, cycle_spend_start, money_budget)` is a pure formatter returning `"Cycle spend: $X.XXXX / $Y.YY"` or `None`; `_print_cycle_spend(...)` is the I/O wrapper. Called after scan, plan, and execute phases; also added to the final Done Panel alongside `Spend today:`. No-op when no money cap is set (avoids clutter on time-only / uncapped runs).

**Meta-note:** this item was the **first self-dogfood test of the planning-quality hypothesis** — a hand-crafted sharp work item (per-file acceptance + verification mandate + explicit out-of-scope list) was piped straight to a coder agent, bypassing scan+plan. The coder produced mergeable code on the first pass: clean pure/IO split, respected every out-of-scope rule, and fed back a useful observation about the dry-run code path. Evidence that sharp plans → good output from Claude Code, supporting the [7] direction of investing in the planner contract rather than coder-side guards.

**Regression tests:** `tests/test_work_budget.py` — 3 new cases covering format-with-cap, None-without-cap, zero-spend-at-start. Pure formatter tests matching the existing pattern in the file.

### [4] `--budget $X` is daily-total, not per-run (surprising semantics) — FIXED (this branch)

**Symptom:** passing `--budget "15m,$2"` on a fresh cycle failed immediately with `session money budget reached ($2.12 / $2.00)` because earlier runs today had already spent $2.12. No work was attempted.

**Root cause:** `src/sentinel/cli/work_cmd.py:742` — single-cycle `_check_all_budgets` compared `--budget $X` against `budget.today_spent_usd` (the daily running total), not against this run's spend. So the flag behaved as a second daily cap, not a per-run cap. Loop mode already tracked session-relative spend correctly (`session_spent = current - session_spend_start`); single-cycle mode didn't.

**Impact (production):** a user expecting `--budget $5` to mean "spend up to $5 on this run" was blocked unexpectedly once any accumulated daily spend neared the flag. The error message doubled down on the confusion ("session money budget reached"). `daily_limit_usd` in config.toml already serves the daily-cap purpose; the `--budget` flag is per-run.

**Fix:** `_run_single_cycle` snapshots `today_spent_usd` at cycle start into `cycle_spend_start` (mirrors loop mode). `_check_all_budgets` takes `cycle_spend_start` and compares `(today_spent_usd - cycle_spend_start) >= money_budget`. Error message now reports the cycle delta and says "per-run budget reached." `--budget` Click help text in `main.py` clarifies per-run vs daily semantics.

**Regression tests:** `tests/test_work_budget.py` — 5 cases covering: cycle allowed when daily total already exceeds cap (the original bug), cycle blocked when delta hits cap, daily limit blocks independently, error message says per-run, no money cap means no per-run gate.

### [3] Same missing-config class affects pre-push hook — FIXED (this branch)

**Symptom:** after the commit path was fixed, the next cycle got further — coder wrote code, commit landed, reviewer approved — then failed at `git push` with the exact same signature: `No .pre-commit-config.yaml file was found`. The global `pre-commit` installs both a pre-commit and a pre-push hook; fixing one didn't fix the other.

**Why this matters (beyond the immediate fix):** this is the `audit-weak-points.md` pattern in action. Step 1 fixed one instance; step 2 should've audited the whole class (any git operation that fires a hook in a configless repo). Audit now done; shared primitive extracted.

**Fix:** moved detection + retry into `git_ops.py` as `run_git_with_precommit_recovery`. Both `_commit_files` (coder) and `ship_pr` (pr) call it. Future git ops that fire hooks (merge, rebase, etc.) can adopt the same wrapper trivially.

**Guardrail candidate (deferred):** a lint/AST rule that flags any new `run_git(["push", ...])` / `run_git(["commit", ...])` call in `src/sentinel/**` and asks the author to consider `run_git_with_precommit_recovery`. Low priority — grep suffices today.

**Regression tests:** `tests/test_git_ops.py` covers commit retry, push retry, skipped-when-config-present, skipped-when-deleted-from-HEAD, happy-path no-op.

### [2] `uv tool install --force` silently serves stale build from cache — FIXED (this branch)

**Symptom:** after `uv tool install --force --from <source-tree> sentinel`, the installed tool was still 72 lines shorter than the source and missing the committed fix. `--force` replaced the venv but `uv` kept a cached wheel keyed on `(name, version)` — since the version was still `0.2.0`, the cache hit.

**Impact (production):** any user who installs Sentinel via `uv tool install`, then upgrades after a fix, would silently keep running the old code until they bumped the version or cleared the cache. A distribution-gate blocker — "fixed in main" is meaningless if upgrades don't propagate.

**Root cause:** version was hard-coded in two places (`pyproject.toml` and `src/sentinel/__init__.py`, both `"0.2.0"`). Behavior-changing commits did not bump it, so the uv cache key `(name, version)` stayed identical across dozens of commits. This is "derive, don't persist" in action — a hand-maintained version field drifts silently from the source it's supposed to label.

**Fix:** switched to `hatch-vcs` for git-derived versioning. `pyproject.toml` declares `dynamic = ["version"]` and `[tool.hatch.version] source = "vcs"`; `src/sentinel/__init__.py` reads `__version__` from `importlib.metadata.version("sentinel")`. Every commit now produces a unique version like `0.2.1.dev12+g49c9044d0.d20260417`, so the uv cache key always invalidates on source change. Tagged releases stay clean (`0.2.1`). `src/sentinel/_version.py` is gitignored (build-time artifact).

**Verification:** `uv sync` now installs `sentinel==0.2.1.dev12+g49c9044d0.d20260417` rather than `0.2.0`. `uv build --wheel` produces `sentinel-0.2.1.dev12+g49c9044d0.d20260417-py3-none-any.whl` — unique filename per commit, unique cache key, no staleness.

**Regression tests:** `tests/test_version.py` — 5 cases pinning: `__version__` is non-empty, PEP 440-shaped, `pyproject.toml` does NOT hard-code version, hatch-vcs is wired as the version source, `__init__.py` reads from `importlib.metadata` (no hard-coded literal). A future revert to a static version fails these tests.
