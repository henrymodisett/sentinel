## Open findings

### [8] Coder↔Reviewer iteration loop — FIXED (this branch)

**Rationale:** the whole point of Sentinel is to discover impactful work **and ship it**. Without an iteration loop, the first reviewer verdict is final — a coder producing partial/buggy work gets rejected once and the whole effort rots on an unmerged branch. Cycle 3 demonstrated this with 2/2 items rejected on fixable findings. The pipeline was "find good work, then fail to ship it."

**Implementation:** `_iterate_coder_reviewer` in `work_cmd.py` wraps the initial review. On non-approved verdict, re-invokes `Coder.execute` with `review_feedback=ReviewResult` so the coder gets the REVISE_PROMPT (explicit blocking issues, "refine don't rewrite" framing, preserved scope rules). Re-runs reviewer. Caps at `MAX_CODER_ITERATIONS = 3`.

**Termination conditions** (priority order):
1. `review.verdict == 'approved'` → proceed to verifier + ship
2. `iterations >= MAX` → stop, final verdict reported to cycle summary
3. **No-progress guard:** if two consecutive reviews produce the same (order-insensitive, whitespace-normalized) `blocking_issues` set, the coder isn't responding to feedback — stop burning budget
4. `exec_result.status == 'failed'` mid-loop → bail, surface the error

**Integration with Touchstone:** clean. The iteration loop is internal to Sentinel (coder ↔ sentinel's LLM reviewer, cheap + fast). Touchstone's Codex pre-push review still runs at push time on the final cleaned-up diff. Boundary from CLAUDE.md holds: no imports, no subprocess calls. Future: when Codex blocks a push, its findings could feed into another iteration pass — the `review_feedback` seam is abstract enough.

**Tests:** `tests/test_coder_iteration.py` — 10 cases covering approve-on-2nd-pass, approve-on-3rd, max-iterations-cap, no-progress-early-stop, order/whitespace-insensitive comparison, coder-failure-breaks-loop, REVISE_PROMPT rendering.

**Per-iteration cost:** each revise pass costs ~1 coder call + 1 reviewer call (~$0.50-$0.60 combined with current routing). Worst case: 3 iterations burns ~$1.80 per item. The no-progress guard cuts this by ~33% when the coder is stuck. This is still much cheaper than a merged PR + later revert, which is the alternative.

### [7] End-to-end pipeline works; coder quality is the gate (afternoon run 3)

**Observation:** cycle 3 completed cleanly — 2 work items processed, each through coder → reviewer. Reviewer correctly rejected both (item 1: coder only addressed 2 of 10 target files and hardcoded image dimensions; item 2: coder emitted invalid CSS `@src/app/components/MediaItem.tsx (prefers-reduced-motion: reduce)` instead of `@media (...)`).

**What this proves:** harness and safety rail are sound. Reviewer catches real bugs; no bad PRs shipped. Scan + plan + execute + review + gate all compose.

**What it exposes:** the bottleneck for the "autonomous PR factory" vision is **coder output quality**, not the harness. Two independent failure modes:
- **Scope discipline** — coder declared done after touching 2/10 files named in the work item's "files likely to be touched"
- **Basic syntax correctness** — coder emitted invalid CSS that the claude-sonnet-4-6 model should have caught in its own test pass

**Proposed directions (not blocking for ship, but shape v0.2):**
- Have the coder prompt mandate a "acceptance checklist" that enumerates each file in "files likely to be touched" and verifies each was modified (or explicitly justified as not needing change).
- Run the target project's lint/format hook inside the coder's verification step, before the reviewer sees the diff. A CSS syntax error should fail `next lint` / `prettier` locally and force the coder to retry.
- Consider a "syntax sanity" reviewer lens that runs cheap static checks (ESLint, CSS parser, etc.) before invoking the LLM reviewer — would have caught the `@src/app/...` bug for ~free.

**Has NOT been proven end-to-end yet:** push-recovery path in a real cycle. Neither rejected item tried to push. Unit tests cover it; need a cycle where an item passes review to prove the full happy path.

### [4] `--budget $X` is daily-total, not per-run (surprising semantics)

**Symptom:** passing `--budget "15m,$2"` on a fresh cycle failed immediately with `session money budget reached ($2.12 / $2.00)` because earlier runs today had already spent $2.12. No work was attempted.

**Root cause:** `src/sentinel/cli/work_cmd.py:742` — single-cycle `_check_budget_gate` compares `--budget $X` against `budget.today_spent_usd` (the daily running total), not against this run's spend. So the flag behaves as a second daily cap, not a per-run cap. Loop mode already tracks session-relative spend correctly (`session_spent = current - session_spend_start` at line 1112); single-cycle mode doesn't.

**Impact (production):** a user expecting `--budget $5` to mean "spend up to $5 on this run" will be blocked unexpectedly once any accumulated daily spend nears the flag. The error message doubles down on the confusion ("session money budget reached"). `daily_limit_usd` in config.toml already serves the daily-cap purpose; the `--budget` flag should be per-run.

**Proposed fix:** single-cycle mode should snapshot `today_spent_usd` at start (mirroring loop mode's `session_spend_start`) and compare `(current - snapshot) >= money_budget`. Error message should say "per-run budget reached" not "session money budget." Also update the `--help` text to be explicit.

**Regression test:** simulate a cycle with `today_spent_usd > money_budget` at start; cycle must still run (and track its own spend against the flag).

### [3] Same missing-config class affects pre-push hook — FIXED (this branch)

**Symptom:** after the commit path was fixed, the next cycle got further — coder wrote code, commit landed, reviewer approved — then failed at `git push` with the exact same signature: `No .pre-commit-config.yaml file was found`. The global `pre-commit` installs both a pre-commit and a pre-push hook; fixing one didn't fix the other.

**Why this matters (beyond the immediate fix):** this is the `audit-weak-points.md` pattern in action. Step 1 fixed one instance; step 2 should've audited the whole class (any git operation that fires a hook in a configless repo). Audit now done; shared primitive extracted.

**Fix:** moved detection + retry into `git_ops.py` as `run_git_with_precommit_recovery`. Both `_commit_files` (coder) and `ship_pr` (pr) call it. Future git ops that fire hooks (merge, rebase, etc.) can adopt the same wrapper trivially.

**Guardrail candidate (deferred):** a lint/AST rule that flags any new `run_git(["push", ...])` / `run_git(["commit", ...])` call in `src/sentinel/**` and asks the author to consider `run_git_with_precommit_recovery`. Low priority — grep suffices today.

**Regression tests:** `tests/test_git_ops.py` covers commit retry, push retry, skipped-when-config-present, skipped-when-deleted-from-HEAD, happy-path no-op.

### [2] `uv tool install --force` silently serves stale build from cache

**Symptom:** after `uv tool install --force --from <source-tree> sentinel`, the installed tool was still 72 lines shorter than the source and missing the committed fix. `--force` replaced the venv but `uv` kept a cached wheel keyed on `(name, version)` — since the version was still `0.2.0`, the cache hit.

**Impact (production):** any user who installs Sentinel via `uv tool install`, then upgrades after a fix, may silently keep running the old code until they bump the version or clear the cache. This is a distribution-gate blocker — "fixed in main" is meaningless if upgrades don't propagate.

**Workaround in the moment:** `uv tool uninstall sentinel && uv cache clean sentinel && uv tool install --reinstall .` did the right thing.

**Proposed fix:** every PR that changes behavior must also bump the version in `src/sentinel/__init__.py` (dev-build suffix is fine — `0.2.0.dev0+<sha>`), so the uv cache key invalidates on content change. Alternatively: document `uv cache clean sentinel` in upgrade instructions. Preferred is the version bump since it's automatic.

**Regression guard candidate:** a test or release script that refuses to ship unless the version changed vs the previous main tip.
